# Raw Data — Instacart Market Basket Analysis

The full CSVs in this folder are **gitignored** because the largest file is 577 MB (well past GitHub's 100 MB per-file limit). This README documents the schema so the repository is self-explanatory; the [`sample/`](sample/) subfolder ships a 100-order, referentially-consistent slice (~46 KB total) so the pipeline can be smoke-tested without downloading the full dataset.

> Thai version: [README_TH.md](README_TH.md)

## Source

- **Dataset**: Instacart Market Basket Analysis
- **Provider**: Instacart, released for a Kaggle competition (2017)
- **License**: Custom — research / non-commercial use, see Kaggle terms
- **Download**: <https://www.kaggle.com/competitions/instacart-market-basket-analysis/data>

After downloading, unzip all six CSVs directly into this folder so the layout matches the schemas below.

## Why this dataset

The Data Engineer role this project targets is at Gosoft (CP All / 7-Eleven group). Instacart is the closest large public dataset to retail / FMCG transactions: real product hierarchy (department → aisle → product), real basket structure, and real reorder behaviour — the same shape as a convenience-store POS feed.

## Schema

### `aisles.csv`
| Column | Type | Note |
|---|---|---|
| `aisle_id` | int | PK |
| `aisle` | string | aisle name, e.g. `prepared soups salads` |

### `departments.csv`
| Column | Type | Note |
|---|---|---|
| `department_id` | int | PK |
| `department` | string | e.g. `frozen`, `produce`, `bakery` |

### `products.csv`
| Column | Type | Note |
|---|---|---|
| `product_id` | int | PK |
| `product_name` | string | |
| `aisle_id` | int | FK → `aisles.aisle_id` |
| `department_id` | int | FK → `departments.department_id` |

### `orders.csv` (one row per order)
| Column | Type | Note |
|---|---|---|
| `order_id` | int | PK |
| `user_id` | int | customer |
| `eval_set` | string | `prior` / `train` / `test` — which split the order belongs to |
| `order_number` | int | 1-indexed sequence of orders for that user |
| `order_dow` | int | day of week (0 = Sunday) |
| `order_hour_of_day` | int | 0–23 |
| `days_since_prior_order` | float | nullable on the user's first order |

### `order_products__prior.csv` and `order_products__train.csv` (one row per line item)
| Column | Type | Note |
|---|---|---|
| `order_id` | int | FK → `orders.order_id` |
| `product_id` | int | FK → `products.product_id` |
| `add_to_cart_order` | int | 1-indexed position in basket |
| `reordered` | int | 1 if the user has bought this product before, else 0 |

## Splitting the large file for Databricks

`order_products__prior.csv` is 577 MB — too large for Databricks Community Edition. Use [`split_prior.py`](split_prior.py) to reduce it before uploading:

```bash
# Keep 30% of rows → order_products__prior_30pct.csv (~173 MB)
python data/raw/split_prior.py

# Keep a custom percentage, e.g. 50%
python data/raw/split_prior.py --pct 50

# Split into two equal halves → _1.csv / _2.csv (~289 MB each)
python data/raw/split_prior.py --split

# Split into two files, 40% each
python data/raw/split_prior.py --split --pct 40
```

## Sample data

The [`sample/`](sample/) subfolder is generated by [`generate_samples.py`](generate_samples.py). It takes the first 100 orders from `orders.csv` and filters every other table down to rows that reference those orders, so foreign keys are intact end-to-end.

| File | Sample rows |
|---|---:|
| `sample/orders.csv` | 100 |
| `sample/order_products__train.csv` | 104 |
| `sample/order_products__prior.csv` | 1,011 |
| `sample/products.csv` | 505 |
| `sample/aisles.csv` | 91 |
| `sample/departments.csv` | 20 |

To regenerate after replacing the raw files:

```bash
python data/raw/generate_samples.py
```
