1. TL;DR — หนึ่งบรรทัดต่อชั้น

  • Bronze = "เก็บไว้แบบเดิมที่สุด" — raw landing zone ที่ replay ได้
  • Silver = "ทำให้ถูกต้องและเชื่อมกันได้" — clean, dedupe, join, enforce schema
  • Gold = "พร้อมใช้ตอบคำถามธุรกิจ" — aggregated marts, 1 mart = 1 stakeholder need
Mental model: Bronze เก็บ "ความจริง" (truth), Silver สร้าง "ความเข้าใจ" (model), Gold ตอบ "คำถาม" (decision)

2. ทำไมต้องแบ่ง 3 ชั้น (ไม่ทำ ETL ชั้นเดียวเลย)

คำถามที่ดีที่ interviewer ชอบถาม. ถ้าเขียน CSV → final dashboard ตรงๆ มันก็ work — แต่จะเจอปัญหา 4 อย่างนี้:

1. Reprocessing

ถ้า business rule ใน Silver เปลี่ยน (เช่น นิยาม "active customer") เราต้องไปดึง raw มาใหม่จากระบบต้นทาง? ไม่ทัน. มี Bronze เก็บ raw immutable ไว้ก่อน → reprocess ได้ทันที

2. Different SLAs

Raw landing โดน upstream ใส่ทุก 5 นาที, dimensions อัปเดตวันละครั้ง, KPI dashboard ใช้สดทุก 15 นาที — แต่ละชั้นมีความถี่/ขนาด/format ที่เหมาะคนละแบบ

3. Quality contracts

Bronze รับทุกอย่าง (รวม dirty rows) — Silver บังคับ schema + dedupe + drop NULL keys — Gold ตรวจ business KPI. แยกชั้น = แยกความรับผิดชอบของ data quality

4. Cost & performance

Gold เป็น aggregated เล็กมาก → query ถูก. ถ้าไม่มี Gold คน analyst จะวิ่ง GROUP BY 32M rows ทุกครั้ง = เปลือง compute

3. Bronze Layer — Raw Fidelity

🟫 Bronze · "เก็บความจริงไว้ก่อน"

Purpose
Landing zone ที่ เก็บข้อมูลดิบจากต้นทางตามจริง — minimal transformation, preserve forensic lineage
Inputs
CSV / JSON / API / CDC stream จาก source system
Outputs (this project)
Parquet (local) / Delta (Databricks) — bronze_orders, bronze_products, bronze_aisles, bronze_departments, bronze_order_products_prior, bronze_order_products_train
Transforms allowed
เปลี่ยน format อย่างเดียว (CSV → Parquet/Delta), แปะ ingested_at ได้, ห้าม filter rows / change types / join
Schema policy
schema-on-read OK — ใส่ทุก column ตาม source แม้แต่ที่ไม่ใช้
Idempotent?
Yes — rerun ได้โดยไม่กลัว double-count (overwrite by partition / merge by source key)

Why Bronze ≠ ก็คือ raw CSV ตรงๆ?

  • Format upgrade — Parquet/Delta อ่านเร็วกว่า CSV เป็น 10×, push down predicate ได้, schema strict
  • Single source of truth — analytic layer ห้ามไปแตะ raw CSV ที่ shared drive โดยตรง (เปลี่ยน folder ที, pipeline พังทั้งบ้าน)
  • Replay safety — ถ้า Silver มี bug, แก้แล้วรัน 02_silver.py ใหม่ได้จาก Bronze ทันที โดยไม่ต้องไปขอ data จาก source อีก
This project: pipelines/01_bronze.py อ่าน CSV จาก data/raw/ เขียนเป็น Parquet ลง data/bronze/ โดย ไม่ filter, ไม่ join, ไม่แปลง type

4. Silver Layer — Cleaned + Conformed

⬜ Silver · "ทำให้ถูกต้องและเชื่อมกันได้"

Purpose
Business entities ที่สะอาด, deduped, joined, มี contract — เป็น input ที่ทุก downstream ใช้ร่วมกัน
Inputs
Bronze tables ทั้งหมด
Outputs (this project)
Delta — silver_dim_products, silver_dim_users, silver_fact_order_items
Transforms
type cast, trim, dedupe, drop NULL keys, join lookup tables, conform column names, partitioning
Schema policy
schema-on-write strict — Silver บังคับ contract, ถ้า Bronze มี column ใหม่ ต้องตัดสินใจก่อน promote
Partitioning (this project)
fact_order_items partition by department (21 ค่า, cardinality พอเหมาะ, predicate pushdown ลด scan ~95%)

สิ่งที่ Silver ต้องทำในโปรเจกต์นี้

  • Conform — รวม order_products__prior + order_products__train ผ่าน UNION ALL (ไม่ใช่ UNION เพราะ Kaggle split การันตีไม่มีซ้ำ → skip dedup sort = เร็วกว่า ~2×)
  • Enrich — join products ↔ aisles ↔ departments เป็น dim_products ที่มีทุก context
  • Aggregate to user grain — dim_users = 1 row ต่อ user พร้อม total_orders, avg_days_between_orders
  • Quality gate — filter(order_id IS NOT NULL AND product_id IS NOT NULL) + dropDuplicates(order_id, product_id) ก่อน promote
Why Delta (not Parquet) ที่ชั้นนี้:
• ACID transactions — multiple jobs เขียนพร้อมกันไม่พัง
• Time travel — query VERSION AS OF n ย้อนกลับได้
• Schema evolution — เพิ่ม column โดยไม่ rewrite ทั้ง table
• Z-Order / OPTIMIZE — compact small files หลังหลายๆ rerun

5. Gold Layer — Business Marts

🟨 Gold · "พร้อมใช้ตอบคำถามธุรกิจ"

Purpose
Aggregated marts ที่ตอบคำถาม stakeholder เฉพาะกลุ่ม — 1 mart = 1 ผู้ใช้ / 1 use case
Inputs
Silver dims + facts
Outputs (this project)
4 Delta marts:
  • gold_dept_kpi — สำหรับ Category Manager (รายแผนก)
  • gold_sku_performance — สำหรับ Merchandising (รายสินค้า)
  • gold_hourly_demand — สำหรับ Operations / workforce planning (DOW × hour)
  • gold_customer_segment — สำหรับ CRM / Marketing (VIP / Loyal / Regular / New)
Transforms
GROUP BY, window functions, business rules (เช่น total_orders ≥ 50 → "VIP")
Grain
หยาบกว่า Silver มาก — Gold มักลดจาก 32M rows เหลือ 21–50K rows → query ใต้วินาที

หลัก "1 mart = 1 stakeholder need"

อย่าทำ Gold table เดียวที่ใหญ่และตอบทุกคำถาม. แยกให้ แต่ละ mart มีผู้ใช้ที่ชัด, มี KPI ที่ชัด, มี SLA ที่ชัด:

Martใครใช้คำถามที่ตอบGrain
gold_dept_kpiCategory Manager"แผนกไหน reorder rate สูงสุด?"1 row / department
gold_sku_performanceMerchandising"SKU ไหนขายดี + ลูกค้าซื้อซ้ำเยอะ?"1 row / product
gold_hourly_demandOperations"peak hour อยู่ตอนไหน วางคนยังไง?"DOW × hour × dept
gold_customer_segmentCRM / Marketing"VIP มีกี่คน, ส่ง campaign ไหนกับใคร?"1 row / user
This project: pipelines/03_gold.py + pipelines/04_serve.py สร้าง 4 marts แล้ว push ลง PostgreSQL ในตาราง kpi_* เพื่อให้ BI / dashboard ดึงต่อได้โดยไม่ต้องแตะ Spark

8. Interview takeaway

คำถามที่จะโดน: "ทำไมต้อง 3 ชั้น ไม่ทำ raw → dashboard ตรงๆ?"

คำตอบ 30 วินาที:
Medallion แยกความรับผิดชอบของ data quality และ cost. Bronze เก็บ raw ให้ replay ได้ — Silver บังคับ contract และ dedupe ให้ทุก downstream ใช้ตรงกัน — Gold pre-aggregate ให้ analyst ไม่ต้อง scan 32M rows ทุก query. ถ้า business rule เปลี่ยน, แก้ที่ Silver / Gold รัน rerun ได้ทันที โดยไม่ต้องไปขอ raw จาก source ใหม่.

คำถามต่อยอด

  • "Bronze ใช้ Parquet หรือ Delta?" → ขึ้นอยู่กับ writer pattern. ถ้า append-only ครั้งเดียวจบ → Parquet ก็พอ. ถ้ามี MERGE / late-arriving / multiple writers → Delta จะปลอดภัยกว่าเพราะมี ACID
  • "Silver ทำไมต้อง dedupe?" → เพราะ source อาจส่งซ้ำ (retry, replay, dual-writer). ถ้า Gold ใช้ Silver โดย assume unique แล้วซ้ำ → KPI ผิดเงียบๆ
  • "Gold ควรมีกี่ marts?" → ตามจำนวน stakeholder use case ที่มี ownership ชัด. mart ที่ไม่มีคนใช้คือ technical debt — ลบทิ้ง
  • "ถ้าวันหนึ่ง Bronze schema เปลี่ยน?" → Silver job ควร fail loud (schema enforcement), ไม่ใช่ silent-drop column. แก้ที่ Silver, rerun, Gold downstream ได้ค่าใหม่
