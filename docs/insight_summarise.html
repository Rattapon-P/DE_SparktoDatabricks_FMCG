Why this dataset for a Gosoft (CP All / 7-Eleven) interview

Interview line: "เลือก Instacart เพราะ schema กับ business question ใกล้ retail ที่สุดในบรรดา public dataset — ผมไม่ต้องสมมุติ business context, เปิดข้อมูลแล้วเห็นเลยว่า dept reorder rate ต่างกันยังไง"

3. BRONZE raw landing

What it is

Bronze = ข้อมูลดิบจาก source ที่เปลี่ยนแค่ format (CSV → Parquet) เพื่อให้ query เร็วขึ้น, ไม่แก้ข้อมูล, ไม่ rename column, ไม่ทำ business logic. Source data ทั้ง 6 ไฟล์ถูก ingest เข้า /data/bronze/ เป็น Parquet 1-to-1.

Why Parquet (ไม่ใช่ CSV)

PropertyCSV (raw)Parquet (Bronze)
Layoutrow-basedcolumnar
Compressionplain textSnappy / Zstd built-in (~5–10× smaller)
Schemainfer ทุกครั้งที่อ่านเก็บ schema ในไฟล์ → อ่านเร็ว, type-safe
Predicate pushdownไม่มี (ต้องอ่านทุก row)มี — Spark filter ที่ scan เลย
Column pruningไม่มีมี — เลือก 2 column จาก 7 → อ่านแค่ 2
Why: 32M-row CSV scan ทุกครั้ง = ช้ามาก. Parquet ลดทั้ง I/O และ CPU เพราะอ่านเฉพาะ column + filter ที่ใช้.
How to apply (interview): "ผมแปลง CSV → Parquet ที่ Bronze เพราะ Parquet เป็น columnar, มี predicate pushdown และ schema metadata → query downstream เร็วขึ้นโดยไม่ต้อง re-parse CSV"

Why "raw, no transform"

หลัก Medallion: Bronze ต้องสะท้อน source 1-to-1 เพื่อให้ replayability ได้ — ถ้า business logic ที่ Silver/Gold ผิด, ย้อนแก้ได้โดยไม่ต้องโหลดจาก source ใหม่ (ซึ่งอาจจะเปลี่ยนไปแล้ว). เป็นเหตุผลเดียวกับ Stage / Landing zone ของ Azure Data Factory ที่คุณคุ้น.

เทียบกับ ADF (ประสบการณ์เดิมของคุณ):
Bronze ในที่นี้ ≈ Stage หรือ Landing container ใน ADLS ที่ ADF copy activity เขียนไฟล์ดิบลงไป. ต่างกันแค่ format = Parquet (compressed) แทน CSV/JSON.

4. SILVER analyst-ready (Delta Lake)

What it is

Silver = ข้อมูลที่ทำความสะอาด + denormalize + audit แล้ว, schema เสถียร, analyst เปิดใช้ได้ทันที. ใน notebook 03_silver_file.ipynb สร้าง 3 ตาราง:

TableRowsSource (Bronze)Transformation

4.1 dim_products — denormalized product dimension

products  ─┐
aisles    ─┼─→ broadcast join (small) ─→ dim_products (50K)
departments┘

Why this shape

  • Denormalize 3 tables → 1: analyst ไม่ต้อง join 3 ตารางทุก query (3NF → star-schema dimension)
  • broadcast join: aisles (2.5KB) และ departments (270B) เล็กกว่า 10MB threshold (default spark.sql.autoBroadcastJoinThreshold) → Spark ส่งทั้งตารางไปทุก executor แทน shuffle (ดู Q&A ด้านล่าง)
  • trim(product_name): data hygiene — ป้องกัน " Banana" ≠ "Banana"
  • dropDuplicates(["product_id"]): PK guarantee — ถ้า upstream ส่งซ้ำมา ก็ยังเหลือ row เดียว
  • silver_loaded_at: audit column — ตอบคำถาม "row นี้เข้ามาเมื่อไหร่" → ใช้ debug + lineage

4.2 dim_users — derived user dimension

orders ─→ groupBy(user_id) ─→ agg(count, min, max, avg) ─→ dim_users (200K)

Why this shape

Source ไม่มี users.csv — แต่ user_id อยู่ใน orders table. การสร้าง dim_users คือการ derive dimension จาก fact (pattern ที่พบบ่อยเมื่อ source ไม่มี master data).

  • total_orders = countDistinct(order_id) → กี่ครั้งที่ user สั่ง
  • avg_days_between_orders = avg(days_since_prior_order) → ความถี่ใช้ทำ segmentation ที่ Gold

4.3 fact_order_items — the big fact

order_products_prior (32M) ─┐
orders (3.4M)               ├─→ filter null + dedup + join ─→ partitionBy(department) ─→ fact_order_items
dim_products (50K, broadcast)┘                                                              (32M, 21 partitions)

Why this shape

  • filter null + dedup ก่อน join: ลด row ที่จะ join → ลด shuffle. Pattern "filter early" / "predicate pushdown"
  • broadcast dim_products (50K · ~2MB): ตารางเล็ก, broadcast เร็วกว่า sort-merge join. คุณ test แล้วใน notebook 02 — broadcast 0.21s vs regular 0.32s
  • partitionBy("department"): query downstream เกือบทุกอันมี WHERE department = 'produce' หรือ groupBy(department). Partition pruning จะข้ามไฟล์ที่ไม่ตรงทันที (ลด I/O 90%+)
  • 21 partitions: ตรงกับ 21 departments — ขนาดเหมาะ (rule of thumb: partition ≥ 128MB, ไม่เล็กจนเกิด small-files problem)

Why Delta Lake (ไม่ใช่ Parquet เปล่า)

Silver/Gold ใช้ Delta ไม่ใช่ Parquet เพราะได้ feature เพิ่ม:

FeatureParquetDelta Lake
ACID transactions✗✓ — write ครึ่งทาง crash, reader ไม่เห็น dirty data
Schema enforcement✗✓ — กัน schema drift (column type เพี้ยน)
Time travel✗✓ — VERSION AS OF 0
UPDATE / DELETE / MERGE✗ (rewrite ทั้งไฟล์)✓ — MERGE INTO สำหรับ SCD2, CDC
Compaction (OPTIMIZE)manual✓ built-in
Z-ORDER (multi-dim cluster)✗✓
Interview line: "Parquet เป็น file format. Delta = Parquet + transaction log (_delta_log/) → ได้ ACID + time travel + MERGE. ในการ build pipeline ที่ต้อง upsert / replay / audit การมี Delta ลด complexity มาก เทียบกับ Parquet ที่ต้อง overwrite ทั้ง partition"

ใน notebook คุณ verify ด้วย DeltaTable.forPath(...).history() ที่แสดง version 0 · WRITE · 32,434,489 rows — นี่คือ transaction log ที่ Parquet ไม่มี.

6. SERVING PostgreSQL via JDBC

Why a separate serving layer

Delta อ่านได้เฉพาะใน Spark (หรือ Databricks). BI tool (Tableau / PowerBI / Looker) ไม่ได้พูด Delta โดยตรง — มันพูด SQL via JDBC. ดังนั้น Gold marts (ที่ aggregate แล้วเล็กพอ) ถูก push ไป Postgres ผ่าน JDBC.

Postgres tableSource martTrick
kpi_deptdept_kpifull push (21 rows)
kpi_sku_top1000sku_performancetop 1000 only (50K → 1K)
kpi_hourly_demandhourly_demandfull push (3.5K rows)
kpi_customer_segmentcustomer_segmentsummary by segment×band (200K → 11)
Why subset before JDBC: JDBC write เป็น row-by-row insert (ไม่ใช่ bulk parquet). ส่ง 200K rows = ช้า + ใช้ Postgres connection limit เกินจำเป็น. เลือก top 1000 SKU + summary segment เพราะ BI dashboard ส่วนใหญ่ใช้ top-N + summary, ไม่ใช่ raw user list.
How to apply: "ใน production จะใช้ COPY / batch insert + write only delta (rows ที่เพิ่ง update). JDBC สำหรับ batch ทุกคืน, full snapshot ขนาดเล็ก = OK"

Setup ที่คุณทำ

  • Download postgresql-42.7.0.jar ผ่าน !wget ใน notebook
  • Pass jar ผ่าน spark.jars + spark.driver.extraClassPath + spark.executor.extraClassPath (ทั้ง driver และ executor ต้องเห็น jar)
  • ใช้ spark-jdbc + mode("overwrite") — ทุกคืน rebuild เต็ม

7. Cross-layer design decisions (interview gold)

Decision 1 · Why 3 layers, not 1?

Single-stage = read CSV → join → push to Postgres. ไม่มี checkpoint → bug ที่ Gold ต้อง re-read 720MB CSV.
3 layers → bug ที่ Gold rebuild แค่ Gold (ไม่กี่นาที), bug ที่ Silver rebuild Silver+Gold. Checkpoint ระหว่างชั้น = isolation + replay.

Decision 2 · Why partition fact by department, not user_id?

21 departments = balanced partition (~1.5M rows ต่อ partition). User_id 200K → 200K partitions = small-files problem. Choose partition column ตาม query pattern + cardinality ที่เหมาะ (10–10,000 unique values).

Decision 3 · broadcast join เมื่อไหร่?

Right-side < 10MB (default autoBroadcastJoinThreshold). aisles (2.5KB), departments (270B), products (2MB), dim_products (~2MB) — broadcast หมด. Order_products (577MB) ต้อง sort-merge.

Decision 4 · ทำไมไม่ใช้ Spark SQL ทั้งหมด?

DataFrame API + functions library (F.col, F.broadcast) ให้ type checking + IDE completion + control fine-grained join hint. SQL อ่านง่ายกว่า แต่ debug ลำบาก. ในงานจริง mix ทั้งสอง: prototype ด้วย SQL, productionize ด้วย DataFrame API.

8. Interview Q&A — สิ่งที่ interviewer จะถามใน "ทุก step"

About data understanding (DE ที่เข้าใจข้อมูล)

Q: Dataset ที่ใช้คืออะไร ทำไมเลือกอันนี้?
Instacart Market Basket จาก Kaggle 2017. เลือกเพราะ schema กับ business question ใกล้ retail/FMCG ที่สุด — มี product hierarchy 3 ชั้น, basket data, reorder signal, temporal pattern. ตรงกับ POS feed ของ 7-Eleven ที่ Gosoft ดูแล.
Q: order_products__prior กับ order_products__train ต่างกันยังไง?
เป็นการ split สำหรับ Kaggle ML competition: prior = order ก่อนหน้า (ใช้ฝึก), train = order ล่าสุดของลูกค้า (target predict ว่าซื้ออะไร). ใน DE pipeline ผมใช้ prior เท่านั้น เพราะอยากดูพฤติกรรมจริงของ order set ใหญ่ (32M rows) ไม่ใช่ ML target.
Q: days_since_prior_order มี null ตั้ง 200K rows ใน orders — แปลว่าอะไร?
Null คือ order แรกของ user (ไม่มี prior). 200K ≈ จำนวน user ทั้งหมด — ตรงกัน, ไม่ใช่ data quality issue. ตอนทำ Silver dim_users ใช้ avg(days_since_prior_order) — Spark ignore null อัตโนมัติ ใน aggregate functions, จึงไม่ต้อง filter.

About Spark / PySpark

Q: Why Spark over traditional ETL (SSIS / ADF copy activity)?
Spark = distributed in-memory + lazy evaluation + Catalyst optimizer. 32M rows × join × group ทำ in-memory ใน minutes. ETL traditional ส่วนใหญ่ row-by-row หรือ pipe via SQL Server engine — bottleneck ที่ disk I/O. ทั้งคู่ทำ business logic ได้, แต่ Spark scale ที่ TB-PB ส่วน traditional cap ที่ GB.
Q: RDD vs DataFrame vs Dataset?
RDD — low-level, immutable distributed collection ของ object. ไม่มี schema, ไม่มี Catalyst optimizer. ใช้เฉพาะ unstructured data หรือ custom partitioner.
DataFrame — RDD + schema (ตาราง). มี Catalyst (logical → physical plan optimization) + Tungsten (off-heap memory). 90% ของงาน DE ใช้อันนี้.
Dataset — DataFrame + compile-time type safety (เฉพาะ Scala/Java). PySpark ไม่มี Dataset API.
Q: Narrow vs wide transformation?
Narrow: 1 input partition → 1 output partition. ไม่มี shuffle. เช่น filter, select, withColumn, map.
Wide: ต้อง shuffle ข้าม executor. เช่น groupBy, join (sort-merge), distinct, orderBy, repartition.
Why it matters: shuffle = network + disk I/O = แพงสุด. ใน pipeline นี้ผมลด shuffle ด้วย: filter early, broadcast join (lookup 5 ตาราง), partition-by-department ที่ Silver (groupBy(dept) ภายหลังกลายเป็น narrow).
Q: How does partitioning work in Spark?
Spark แบ่งข้อมูลเป็น partition (ค่า default = ขนาด HDFS block / ค่า spark.sql.shuffle.partitions). 1 partition = 1 task ที่ executor ทำขนาน.
2 ระดับ: (1) memory partition — ตอน read/shuffle, (2) storage partition — partitionBy(col) ที่ write → directory layout เช่น department=produce/.
Storage partition สำคัญเพราะ Spark ทำ partition pruning: WHERE department='produce' → อ่านแค่ 1 ใน 21 directory.
Q: How would you optimize a slow Spark job?
Checklist 6 อย่าง:
1. Read explain plan — มี SortMergeJoin ที่ควรเป็น BroadcastHashJoin ไหม?
2. Filter / select early ก่อน join (predicate + projection pushdown)
3. Broadcast small table < 10MB
4. Check partition skew — ถ้า key ตัวเดียว 80% ของ rows → salting
5. Right number of partitions — ขนาด ~128MB ต่อ partition
6. Cache ถ้าใช้ DataFrame ซ้ำ > 1 ครั้ง
โยงประสบการณ์เดิม: ที่ Feedback180 ผม optimize Tableau dashboard 2.5h → 30min ด้วย same principles — push aggregate ลงใน source DB (Postgres) แทนทำใน Tableau → ลด data scanned + ใช้ index. หลักการคล้ายกัน.

About Delta Lake

Q: What is Delta Lake? Why not just Parquet?
Delta = Parquet + transaction log (_delta_log/ เป็น JSON) + checkpoint file ทุก 10 commits.
Delta ให้: ACID, schema enforcement, time travel, MERGE/UPDATE/DELETE, Z-ORDER, OPTIMIZE.
Parquet เปล่าให้แค่: columnar storage. ถ้า write ครึ่งทาง crash, reader เห็น partial file = corrupt. ถ้าจะ UPDATE 1 row ต้อง rewrite ทั้ง partition.
ใน pipeline นี้ ผมใช้ Parquet ที่ Bronze (raw, append-only) และ Delta ที่ Silver/Gold (ต้อง overwrite + history).
Q: ใน _delta_log มีอะไร?
JSON ทุก commit list "เพิ่มไฟล์ X, ลบไฟล์ Y". Reader อ่าน log ไล่จาก checkpoint ล่าสุด → รู้ว่า version ปัจจุบันมีไฟล์ Parquet ไหนบ้าง. Time travel = อ่าน log ถึง version N แทน latest.

About Architecture (Medallion)

Q: ทำไมต้อง 3 layers? 1 layer ไม่พอเหรอ?
Bronze เก็บ history + replay capability (ถ้า source หาย ก็ rebuild จาก Bronze ได้).
Silver แยก data hygiene (cleansing, dedup) ออกจาก business logic — tester / analyst test schema เสถียรได้โดยไม่กระทบ Bronze.
Gold คือ contract กับ BI — ถ้า dashboard ขึ้น KPI ผิด ก็แก้ที่ Gold ไม่ต้องไปยุ่ง raw.
Trade-off: เพิ่ม latency + storage (3 copies). คุ้มเมื่อ pipeline จะ run ยาวเป็นปี + มีคน > 1 ทีมใช้.
Q: ทำไมไม่เก็บ Gold ใน Postgres ตรงๆ?
Gold เป็น source of truth สำหรับ analytics. Postgres เป็น serving cache. 2 ตัวมี role ต่างกัน:
Gold (Delta) — versioned, time-travel, audit. ML team อ่านได้ตรงผ่าน Spark.
Postgres — fast point query สำหรับ BI dashboard. ไม่ต้องเก็บ history เก่ามาก.
ถ้า BI ผิด → query Gold ย้อน version. ถ้า Postgres หาย → rebuild จาก Gold ใน 1 คำสั่ง.

9. Three senior insights you generated

Insight 1 · DE ต้องเข้าใจข้อมูลไม่ต่างจาก DA หรือ DS

การเลือก partition key (department), การออกแบบ aggregate, การ derive dim_users จาก orders — ทุก decision ต้องรู้ก่อนว่า "คำถามธุรกิจคืออะไร" ถ้า DE มอง pipeline เป็นแค่ "ย้ายข้อมูล" จะออกแบบไม่ตรง use case → analyst ต้อง re-aggregate ทุกครั้ง → cost ทีมรวมพุ่ง.

Use this in interview when asked "What makes a good DE?"

Insight 2 · CTE / WITH ใน SQL = Medallion architecture ในระดับ query

SQL CTE ที่เคยใช้ (Stage 1 → Clean → Aggregate → Final) เป็น Bronze → Silver → Gold ในระดับ in-memory query. หลักการเดียวกัน — แค่ scale ต่างกัน:

  • CTE = single query, in-memory, ตายเมื่อ query จบ
  • Medallion = persistent storage, cross-query, cross-tool, มี history

ดังนั้น "experience SQL 5x optimization" ของคุณที่ Feedback180 = foundation เดียวกันกับ Spark optimization — ต่างที่ engine.

Use this when asked "How does your past experience translate to Spark?"

Insight 3 · Data layer ดี → MLflow ใช้ต่อได้ → ลดเวลา/cost ของทีมรวม

ถ้า Silver clean + schema เสถียร → DS ใช้ทำ feature engineering ได้ทันที (ไม่ต้อง re-clean). ถ้า Gold มี customer_segment, sku_performance → DS train churn / recommendation ได้โดยใช้ feature ที่ DE pre-aggregate ให้แล้ว.

นี่คือเหตุผลที่ Databricks รวม Delta + MLflow + Feature Store ใน platform เดียว — เพราะ data layer คือ foundation ของ ML. DE ที่ดี = enabler ของทีม DS, ไม่ใช่แค่คน upstream.

Use this when asked "How do you collaborate with DS / business teams?"

Quick reference — สิ่งที่ต้องตอบได้ใน 30 วินาที

  1. Dataset = Instacart, Kaggle 2017, ใกล้ 7-Eleven POS
