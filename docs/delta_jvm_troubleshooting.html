1. Symptom

เปิด notebook ใหม่ Checking_before_04.ipynb เพื่ออ่าน Silver Delta table ตรวจ orphan rows ก่อนทำ Gold layer. รันแล้วได้ error ดังนี้:

Py4JJavaError: An error occurred while calling o39.load.
: org.apache.spark.SparkClassNotFoundException: [DATA_SOURCE_NOT_FOUND]
  Failed to find the data source: delta.
Caused by: java.lang.ClassNotFoundException: delta.DefaultSource
สิ่งสับสน: ใน notebook 03_silver_file.ipynb โค้ดแบบเดียวกันรันได้ — ทำไมที่นี่ fail?

2. Attempt #1 — point ไปที่ jar local

pg_jar      = f"{work_dir}/postgresql-42.7.0.jar"
delta_jar   = f"{work_dir}/delta-spark_2.12-3.2.0.jar"   # ← ไม่มีไฟล์จริง
storage_jar = f"{work_dir}/delta-storage-3.2.0.jar"      # ← ไม่มีไฟล์จริง
all_jars    = ",".join([pg_jar, delta_jar, storage_jar])

spark = (SparkSession.builder
         .config("spark.jars", all_jars)
         ...)
ปัญหา: cell แรกของ notebook ทำ !wget เฉพาะ postgresql-42.7.0.jar — ไม่ได้ download Delta jars 2 ตัว → path ใน all_jars ชี้ไปที่ไฟล์ที่ไม่มีอยู่จริง
Lesson 1: Spark spark.jars ไม่ throw error ตอน startup ถ้าไฟล์ไม่มี — มัน silent-skip. กว่าจะรู้คือตอน format("delta") หาคลาสไม่เจอ.
How to apply: หลัง getOrCreate() เสมอ list jars ที่ load จริง — ดูในข้อ 7

3. Attempt #2 — switch ไปใช้ spark.jars.packages

เปลี่ยนเป็น Maven coordinate ให้ Spark download Delta + dependencies ครบในคำสั่งเดียว (เหมือน notebook 03 ที่ work):

spark = (SparkSession.builder
         .config("spark.jars", pg_jar)
         .config("spark.jars.packages", "io.delta:delta-spark_2.12:3.2.0")
         .config("spark.sql.extensions", "io.delta.sql.DeltaSparkSessionExtension")
         .config("spark.sql.catalog.spark_catalog", "org.apache.spark.sql.delta.catalog.DeltaCatalog")
         .getOrCreate())

เพิ่ม SparkSession.builder.getOrCreate().stop() เพื่อ kill session เก่าก่อน. รันใหม่ — ยัง error เดิม.

งงตรง: config มา, stop() ทำแล้ว, ทำไมยัง fail?

5. Root cause — PySpark ↔ JVM lifecycle

Architecture

┌──────────────────────────────────────────────────────────┐
│  Jupyter kernel (Python process)                         │
│  ┌────────────┐         py4j           ┌────────────┐    │
│  │  PySpark   │ ←──── socket ────→     │   JVM      │    │
│  │  (Python)  │                        │  (driver)  │    │
│  └────────────┘                        │  ──────    │    │
│                                        │  classpath │    │
│                                        │  เซ็ตตอน    │    │
│                                        │  spawn     │    │
│                                        │  เท่านั้น    │    │
│                                        └────────────┘    │
└──────────────────────────────────────────────────────────┘

เหตุการณ์ที่เกิดขึ้นจริง (timeline)

  1. รัน Attempt #1 → JVM spawn ครั้งแรก, classpath = [pyspark-internal jars] + [postgresql.jar] (Delta jars path ไม่มีไฟล์ → skip)
  2. format("delta") fail
  3. เขียน Attempt #2, ใส่ spark.jars.packages, เรียก stop(), แล้ว getOrCreate() ใหม่
  4. JVM driver ยังเป็นตัวเดิม (py4j socket ยังเชื่อมอยู่). stop() แค่ shutdown SparkContext (DAGScheduler, Executors) — ไม่ kill JVM
  5. getOrCreate() register spark.jars.packages ลง SparkConf ของ context ใหม่
  6. แต่ Ivy resolution มันรันตอน spark-submit bootstrap เท่านั้น — ไม่ใช่ตอน builder รัน. ดังนั้น Delta jar ไม่ถูก download/load
  7. Classpath ของ JVM ยังเป็นเดิม → format("delta") fail อีก

SparkConf().setAll(...) ใหม่alivealiveเดิมไม่มี

One-line root cause: spark.jars.packages resolve dependencies ผ่าน Ivy ตอน JVM bootstrap เท่านั้น. JVM ที่ค้างอยู่ใน Jupyter kernel ไม่ pick up config ใหม่ — ต้อง kill JVM = restart kernel

7. Verify pattern — เพิ่มทุกครั้งที่ start session

เพิ่ม 3 บรรทัดหลัง getOrCreate() เป็น guard เพื่อ fail-fast แทน fail-late ที่ format("delta"):

loaded = spark.sparkContext._jsc.sc().listJars().toString()
assert "delta-spark" in loaded, f"❌ Delta NOT on classpath. Got: {loaded[:200]}"
print("✅ Delta jar verified on classpath")
Why: error ที่ format("delta") เกิดหลัง pipeline รันไปแล้ว 30 วินาที (count, filter, etc.). assert ตรวจตอน startup = error ภายใน 5 วินาที, debugging cycle เร็วขึ้นมาก
How to apply: ใส่ guard ทุก notebook ที่ใช้ Delta. หลักการ "fail fast at boundary" (boundary = config ↔ classpath)

9. Interview takeaway

คำถามสมมุติ: "ถ้าเจอ ClassNotFoundException: delta.DefaultSource ใน notebook คุณ debug ยังไง?"

คำตอบ (30 วินาที):

  1. เช็ค config registered จริงไหม — spark.sparkContext.getConf().get("spark.jars.packages")
  2. เช็ค jar load จริงไหม — spark.sparkContext._jsc.sc().listJars()
  3. ถ้า config มาแต่ jar ไม่ load = JVM ค้างจาก session ก่อน → restart kernel
  4. เพิ่ม assert "delta-spark" in listJars เป็น guard — fail fast ดีกว่า fail-late

ทำไมเรื่องนี้แสดง depth ของ DE

คน junior จะตอบแค่ "เพิ่ม spark.jars.packages สิ" — ตอบจริงแต่ตื้น. คน senior อธิบาย JVM ↔ py4j ↔ Python lifecycle ได้, รู้ว่า:

  • SparkSession.stop() ≠ kill JVM
  • spark.jars.packages resolve ตอน spark-submit bootstrap, ไม่ใช่ตอน builder
  • Jupyter kernel = 1 Python process = 1 JVM driver per kernel lifetime
  • Verify pattern listJars เป็น defensive coding ที่ทำให้ debug cycle เร็ว

นี่คือคำตอบที่แยก DE ที่ใช้ Spark "เป็น" ออกจาก DE ที่ "เข้าใจ" Spark architecture

โยงประสบการณ์เดิม (Feedback180)

"หลักการ fail-fast guard ที่ผมใช้ตรวจ Delta load มันคือไอเดียเดียวกับการ validate config ตอน app startup ใน FastAPI — ผมเคยเจอเคส env var ขาด, ค้นพบตอน request หล่นแทนที่จะเจอตอน boot. หลังจากนั้นผมใส่ pydantic.BaseSettings validation ทุก service. ที่ Spark ก็เป็นหลักการเดียวกัน — แค่ boundary คือ classpath แทน env var"
